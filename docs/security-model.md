# Security Model

## Encryption Layers

```
Layer 1: TLS 1.3 (WSS/HTTPS)
├── Signaling WebSocket (VPS ↔ Client)
├── REST API (Client ↔ VPS)
└── REST API (Client ↔ PC Gamer, LAN only)

Layer 2: DTLS 1.2 + SRTP (WebRTC)
├── P2P DataChannel (PC Gamer ↔ Remote Client)
├── Mandatory per WebRTC spec — cannot be disabled
└── AES-256-GCM with Perfect Forward Secrecy (ECDHE)

Layer 3: Fingerprint Pinning
├── DTLS certificate SHA-256 hash → embedded in JWT
├── Client verifies fingerprint during SDP handshake
└── MITM impossible even with compromised VPS
```

## Threat Model

### Network Attacks

| Threat | Severity | Mitigation |
|--------|----------|------------|
| MITM on signaling WebSocket | CRITICAL | WSS (TLS 1.3) required for production |
| MITM on WebRTC via VPS | CRITICAL | DTLS fingerprint pinned in JWT |
| P2P traffic sniffing | LOW | DTLS-SRTP mandatory, no disable option |
| STUN spoofing | MEDIUM | Only reveals public IP; DTLS verifies identity |
| TURN relay abuse | HIGH | Auth with temp credentials (1h TTL), 5MB limit per session |
| DoS on signaling server | HIGH | Rate limit: 5 conn/s per IP, 60 msg/min per connection |
| DoS via DataChannel | HIGH | Max 10 requests/min per peer, max 5 peers per node |
| Replay JWT | LOW | JWT has `exp`, `iat`, `jti`. Blacklist revoked tokens. |

### Authentication Attacks

| Threat | Severity | Mitigation |
|--------|----------|------------|
| JWT theft (XSS) | HIGH | HttpOnly cookies for web apps. Short TTL (1h). Refresh tokens. |
| API key compromise | CRITICAL | SHA-256 hash in DB. Immediate revocable. |
| Brute force login | MEDIUM | 5 attempts/15min per IP. Lock after 10 failures. |
| Fake malicious node | CRITICAL | API key pre-approved by admin. Nodes cannot self-register. |
| Node enumeration | MEDIUM | Only authenticated users can list models. Private nodes hidden. |

### Application Attacks

| Threat | Severity | Mitigation |
|--------|----------|------------|
| Prompt injection | MEDIUM | Gemma 4 has built-in guardrails. Input sanitization. |
| Path traversal (multimodal) | CRITICAL | Reject paths with `..`. Whitelist allowed directories. |
| SSRF (image fetch) | HIGH | Only `https://` URLs. Block private IPs (`10.x`, `192.168.x`, `127.x`). |
| Oversized payloads | HIGH | Max 10MB for `/api/chat`, 20MB multimodal. |
| Modelfile injection | MEDIUM | Strict parser. No arbitrary code execution. |
| SQL injection | LOW | Parameterized queries only. |
| Race condition (engine) | HIGH | `asyncio.Lock` per model. Global lock for create/destroy. |

## Key Management

- **API Keys**: Generated by admin on the VPS. SHA-256 hash stored. Plaintext shown once.
- **JWT Secret**: Generate with `openssl rand -hex 32`. Set via `LITERT_SIGNALING_JWT_SECRET` env var.
- **Turn credentials**: Auto-generated, temp credentials (1-hour validity). Not stored.
- **DTLS Fingerprints**: Auto-generated per RTCPeerConnection. Embedded in JWT for verification.

## Audit Logging

All actions are logged to the `audit_log` table:
- Login attempts (success/failure)
- Node registration/removal
- Room creation/closing
- Admin actions (invite creation, key generation)
- Rate limit violations

## Production Checklist

- [ ] TLS certificates via Let's Encrypt (Caddy or nginx)
- [ ] Non-default JWT secret (min 32 bytes random)
- [ ] Rate limiting configured
- [ ] TURN server password set
- [ ] Admin invite codes sent via secure channel
- [ ] API keys distributed securely (never in git)
- [ ] Monitoring and alerting
- [ ] Regular security audits
